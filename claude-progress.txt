## Sessão: 2026-02-17 — Circuit Breaker AI + strict_types + Logging

### Problema Identificado
- Todos os 3 provedores AI (OpenAI, Anthropic, Gemini) com chaves inválidas/demo
- `isAvailable()` checava apenas `!empty($apiKey)`, então chaves inválidas passavam
- `executeWithFallback()` tentava TODOS os providers em cada request → 6+ erros de log por request
- Log flooding: 401 Unauthorized (OpenAI), 400 credit_balance_too_low (Anthropic), 400 API key not valid (Gemini)

### Implementação: Circuit Breaker Pattern
- Adicionado circuit breaker em `AIProviderManager` com TTL de 300s
- Após erro de autenticação/chave/crédito, provider é "tripped" e pulado por 5 minutos
- Persistência em arquivo (`/tmp/ai_provider_circuit_breaker.json`) para cross-request
- Detecção de padrões: 'Unauthorized', 'API key', 'credit balance', 'authentication', etc.
- Métodos públicos: `getCircuitBrokenProviders()`, `resetCircuit()`, `resetAllCircuits()`

### Melhorias de Código
- `declare(strict_types=1)` adicionado em 8 arquivos:
  - `AIProviderManager.php`, `AbstractAIProvider.php`
  - `OpenAIProvider.php`, `ClaudeProvider.php`, `GeminiProvider.php`
  - `GapHunterService.php`, `CompetitorService.php`, `CompetitorAnalysisService.php`
- `GapHunterService`: 3 catch blocks silenciosos substituídos por logging estruturado via StructuredLogService
- `CompetitorService` e `CompetitorAnalysisService`: revisados — código sólido com PDO prepared statements

## Sessão: 2026-02-17 — Correção de 4 Bugs (Segurança + Schema Drift + Info Leak)

### Bugs Corrigidos

#### BUG-1: NotificationService — Data truncated for column 'type'
- **Causa raiz**: Tabela `notifications` criada pelo `scripts/migrate_notifications.php` com schema diferente do que o `NotificationService` espera. Coluna `type` no DB real era ENUM ou VARCHAR curto, e o valor `order_new` não cabia.
- **Fix**: Adicionado auto-repair em `ensureNotificationsTable()` que verifica se colunas existem e se `type` é VARCHAR(50). Se não for, executa ALTER TABLE automaticamente.
- **Arquivo**: `app/Services/NotificationService.php`

#### BUG-2: MonitoringService — Unknown column 'status'
- **Causa raiz**: Tabela `system_monitoring` criada anteriormente sem coluna `status`. O `CREATE TABLE IF NOT EXISTS` não recria tabelas existentes, então a nova coluna nunca foi adicionada.
- **Fix**: Adicionado `SHOW COLUMNS` check + `ALTER TABLE ADD COLUMN status` se coluna não existir.
- **Arquivo**: `app/Services/MonitoringService.php`

#### BUG-3: BaseController — Mensagem de erro exposta em produção
- **Causa raiz**: `withErrorHandling()` concatenava `$e->getMessage()` na resposta JSON para qualquer exceção Throwable, expondo informações internas (paths, queries, nomes de tabelas) em produção.
- **Fix**: Adicionado check de `APP_ENV` — em produção retorna mensagem genérica "Erro interno do servidor".
- **Arquivo**: `app/Controllers/BaseController.php`

#### BUG-4: Database.php — SQL Injection em identifiers
- **Causa raiz**: Métodos `insert()`, `update()`, `delete()`, `select()`, `exists()`, `count()` interpolavam nomes de tabela e coluna diretamente no SQL sem sanitização.
- **Fix**: Adicionado método `sanitizeIdentifier()` com regex whitelist + backtick escaping. Aplicado em todos os 6 métodos CRUD.
- **Arquivo**: `app/Database.php`

### Arquivos modificados

- `app/Database.php` — SQL injection fix
